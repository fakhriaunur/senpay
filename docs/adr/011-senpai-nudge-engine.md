# ADR 011: Senpai-Full Nudge Engine — Rule-Based Financial Nudges

**Status:** Accepted  
**Date:** 2026-05-16  
**Deciders:** Senpay Engineering Team

## Context

Senpai is Senpay's financial assistant. The "Senpai-Full" feature set includes a nudge engine that proactively alerts users about unusual spending patterns, budget exhaustion risks, and spending trends. Requirements:

- Nudges must be **deterministic** — same transaction history → same nudge output (ADR 001 compliance)
- Nudges must be **explainable** — users must understand why they received a nudge
- Nudge computation must be **pure** (no I/O) — core logic testable without mocks
- Nudge severity must be graded (warning, critical, info) for UX prioritization
- Optional LLM-generated tips for additional context, but core logic must not depend on LLM availability
- Feature flag `SENPAI_FULL_ENABLED` gates the entire nudge engine (see ADR 008)

## Decision

The nudge engine uses **pure rule-based algorithms** for core nudge generation, with an optional LLM adapter layer for enriched tips. Core logic is deterministic, requires no training data, and produces explainable output.

### Nudge Rules (Pure Core)

Four rule-based detectors, each a pure function:

#### 1. Velocity Detector

Detects spending spikes relative to user's 7-day rolling average.

```
velocity_ratio = current_spending / max(7d_avg, 1)
if velocity_ratio > 1.5: severity = critical
if velocity_ratio > 1.0: severity = warning
```

#### 2. Trend Detector

Detects sustained spending increases using 3-period moving average.

```
ma = (spending[t-2] + spending[t-1] + spending[t]) / 3
if ma shows 2+ consecutive period-over-period increases AND latest > baseline:
    severity = warning
```

#### 3. Anomaly Detector

Detects statistical outliers using mean + 2× standard deviation over a configurable window (default 30 days).

```
threshold = mean(spending_window) + 2 * stddev(spending_window)
if current_spending > threshold: severity = info
```

#### 4. Exhaustion Detector

Projects budget exhaustion date using linear extrapolation.

```
daily_rate = total_spending / days_elapsed
days_remaining = remaining_budget / daily_rate  // linear extrapolation
if days_remaining < 7 AND budget_used >= 90%: severity = critical
if days_remaining < 14 AND budget_used >= 80%: severity = warning
if budget_used >= 100% (exceeded): severity = critical
```

### Severity Model

| Severity | Meaning | UX Priority |
|----------|---------|-------------|
| `critical` | Immediate action needed (e.g., budget exhausted, 2× normal spending) | Highest — push notification |
| `warning` | Attention recommended (e.g., above-average spending, budget >80%) | Medium — in-app badge |
| `info` | Informational (e.g., mild anomaly, spending trend) | Low — feed item only |

### Nudge Output

```go
type Nudge struct {
    ID       string       // UUID v7
    Rule     NudgeRule    // which detector triggered
    Severity Severity     // warning | critical | info
    Message  string       // human-readable explanation
    Tip      string       // optional LLM-generated tip (empty if disabled)
    Metadata NudgeMeta    // rule-specific data for UX rendering
}
```

### LLM Integration (Optional)

LLM tips are generated by a provider-agnostic adapter (see ADR 012). The nudge engine calls the adapter **after** rule evaluation. On LLM failure, timeout, or disabled state, `Tip` is left empty — the core nudge is still delivered. LLM is additive, not essential.

### Architecture

```
handler (feature flag check)
  → service (orchestrates data fetching from shell)
    → core.AnalyzeNudges(txns, budgets, cfg) → []Nudge   // pure
    → llmAdapter.GenerateTips(ctx, nudges) → nudges        // shell, optional
```

Core function `AnalyzeNudges` is pure: takes transaction history, budget state, and config as parameters. No I/O, no network, no `time.Now()` — deterministic output.

## Alternatives Considered

**ML-based anomaly detection (e.g., isolation forest, LSTM)** — Rejected. Over-engineered for consumer e-wallet nudges. Requires training data, model serving infrastructure, and produces non-deterministic results. Rule-based approach is simpler, more explainable, and sufficient for current requirements.

**Pure LLM-based nudge generation (prompt the LLM with transaction history)** — Rejected. Expensive (cost per nudge), non-deterministic (same input may produce different output), and introduces latency. LLM is better suited as a tip-enhancement layer, not the primary engine.

**External rules engine (e.g., Drools, OpenPolicyAgent)** — Rejected. Adds operational complexity and framework dependency. Go functions are simpler, faster, and easier to test with property-based testing.

## Consequences

**Positive:**

- Deterministic: same transaction history always produces same nudges — testable with table-driven tests
- No training data required — works immediately after deployment
- Explainable: every nudge has a clear rule trigger that can be communicated to users
- Core functions remain pure, following ADR 001 FCIS pattern
- Graceful degradation: if LLM is unavailable, core nudges still work
- Feature flag (`SENPAI_FULL_ENABLED`) provides rollout control per ADR 008

**Negative:**

- Rule thresholds (1.5× velocity, 2σ anomaly, 7-day exhaustion) are hand-tuned and may require iteration based on user feedback
- No personalization: rules are the same for all users (LLM tip layer provides some personalization)
- Statistical anomaly detection requires a window of historical data — new users (<30 days) receive fewer nudges
- Adding new detector types requires code changes (acceptable: detectors are pure functions, trivially extensible)

## Compliance

All nudge detectors must be pure functions in `internal/senpai/core.go`. No I/O in detector logic. LLM integration must be behind the `NudgeLLM` interface (ADR 012). Feature availability gated by `SENPAI_FULL_ENABLED` per ADR 008.
